- [Inbox messages & the notification event bus (send_to_user inbox message store, cross-session Inbox, declared event design, push fan-out)](agent-messages-inbox.md) - How an inbox message reaches the human. Two producers feed one store: (1) agent-authored `send_to_user` MCP calls — the session-blind stdio server only declares the schema, and the PostToolUse hook (which alone knows the session) persists them through the single `store.record_message` writer; (2) a declared notification event bus (`events.py`) over which every non-`send_to_user` system event — an agent blocked on a permission/plan, an external proposal drafted, wiki content drift, a session grade landing, a topic proposed for suppression — is declared once in `REGISTRY` and published through one `emit`, routing to the same writer with a deep-link URL. - [Memory auto-injection & on-demand recall (UserPromptSubmit hook, recall MCP server, /api/memory/recall)](memory-auto-injection.md) - Show how recalled memories reach an agent and why the delivery path is shaped the way it is. - [Memory consolidation (reflect: mechanical pre-pass, agentic dream, lifecycle decay, embed & edges)](memory-consolidation-reflect.md) - Explain `reflect()` — the batch consolidation cycle that turns raw `working` rows into curated `episodic` knowledge — as four ordered stages. (1) A mechanical no-model pre-pass: near-duplicate dedup (`_dedup_merge`, embedding cosine or `difflib` text-ratio fallback), the pre-decay engagement sweep (`_score_pending`), and legacy `kind='digest'` retirement. (2) The single agentic `dream` LLM stage (`_dream`): one `llm.complete` over a bounded evidence pack (`_build_dream_pack`) of pending working rows with their co-retrieval neighbours plus up to `contradiction_budget` suspect same-referent episodic pairs, returning one JSON plan (promote/hold/drop/merge, contradict/obsolete/distinct, synthesize) applied deterministically with per-action validation (`_apply_dream_plan`), blind-promote on model outage. (3) Lifecycle decay: forget-stale (`_forget_stale`), chronically-ignored importance decay reading the soft/hard engagement split (`_decay_chronically_ignored`/`_decay_reason`), and stale-reference flagging with git rename-follow (`_flag_stale_references`). (4) Embed (`_embed_episodic`) + `related`-edge harvest (`_harvest_edges`). `topic_attach.py` lands synthesis clusters on the human-approved `.regin/topics` graph.

- [Memory recall pipeline (FTS + dense + RRF + rerank, quality weighting, topic boost, MMR, edge expansion, query expansion)](memory-recall-pipeline.md) - Explain the read-side ranking stack a `recall()` request flows through in `store.py`, so an agent debugging or tuning recall knows which knob moves which result. `recall` embeds the query once, fuses `_lexical_ids` (FTS5/BM25) and `_dense_ids` (cosine over `memory_embeddings`, with `_lazy_backfill`) via `_rrf`, narrows to `_eligible_rows`, applies the `_gate_lexical_overlap` precision gate (idf-filtered informative tokens), reranks the RRF head with the cross-encoder (`_order_candidates`, sigmoid confidence, `rerank_cap`), multiplies by the bounded `_quality_factor`, softly lifts topic-linked rows (`_apply_topic_boost`), applies optional MMR diversity (`_mmr_select`) and 1-hop edge expansion (`_expand_via_edges`), and reinforces on deliberate recall (`_bump_recall`). `expand.py` is the LLM query rewrite; `evaluate.py` is a pointer to the FTS-only regression harness owned by recall-eval-verification.
