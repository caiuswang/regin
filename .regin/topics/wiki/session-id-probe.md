## Session-id probe (`session-id-probe`)

A small, self-contained answer to a problem with no obvious home: **a Bash command needs to know its own Claude Code session id**, and several regin commands depend on it — `goal preflight --session-id`, `goal feedback --trace-id`, and the `gate` anti-skip checks.

Claude Code (>= ~2.1) exports the live session id into every child process's environment as `CLAUDE_CODE_SESSION_ID`. `lib/session_probe.py::resolve()` reads that variable, treating an absent or empty value as a miss (returns `None`) so callers can map blank to 'omit the flag'. The always-present `regin session-id` subcommand (`cli/commands/session.py`) is a thin wrapper: on a hit it prints the id; on a miss it prints nothing and exits 0, keeping `SID=$(… session-id)` a soft, composable failure. Because the variable is read straight from the environment, a single `SID=$(… session-id)` resolves correctly with no prior step and works through `$(...)` substitution and the full `.venv/bin/python cli/regin.py session-id` interpreter form alike — and it is per-process safe: a concurrent session running a Bash call in the same directory can't clobber it.

The env var is authoritative even for a *child* session (`CLAUDE_CODE_CHILD_SESSION=1`): it carries the child's own id, which is where that context's trace spans land — unlike the background-task output directory, which is named with the *parent* session id. So skills read the id here rather than reconstructing it from a Task tool's output path. Behaviour is pinned by `tests/test_session_probe.py` (env present / absent / exported-empty), and `tests/cli/test_cli_registration.py` guards that the `session-id` command stays registered on the real interpreter — the exact regression (`No such command 'session-id'`) that once broke the goal-verified loop. GitNexus's regin index predates this resolver (121 commits behind), so this section rests on the current source alone.
