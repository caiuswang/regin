## Inbox messages & the notification event bus (`agent-messages-inbox`)

Every **inbox message** the human sees flows through one store and one writer. Two kinds of producer feed it: agent-authored `send_to_user` messages, and a declared **notification event bus** for system events the agent didn't type by hand.

**Agent-authored `send_to_user` â€” the capture path is split on purpose.** The stdio MCP server (`lib/agent_messages/mcp_server.py`) is *session-blind*: it never learns which Claude Code session invoked it, so it only declares the typed parameter schema (`type`, `title`, `key`, `links`, `supersedes`) and acknowledges the call. The component that knows the session is regin's PostToolUse hook (`hook_manager/handlers/post_tool_trace.py`, in [[trace-span-capture]]), which reads the tool input off the landed `mcp__*__send_to_user` span and calls the store. This keeps the server import-free and instant, and persistence never depends on the server surviving a DB hiccup.

**One writer.** Every write â€” hook, event bus, web API, or test seed â€” goes through `lib/agent_messages/store.py::record_message`, so supersede-by-`msg_key`, the 16k body cap (`_BODY_MAX = 16_000`), link normalization, and push dispatch (`push.maybe_dispatch`) all happen in exactly one place. Superseding is an in-place upsert on the newest live `(trace_id, msg_key)` row â€” body/title/type/links updated, `version` bumped, `read_at` cleared so the refreshed card re-surfaces as unread. `store` also owns the keyed-card helpers (`live_keyed_message`, `dismiss_keyed`) that supersedable event cards rely on, plus an opt-in retention sweep (`prune_messages` / `_enforce_retention` past `settings.agent_messages.retention_days`, off by default) â€” the only hard delete in a store where `dismissed_at` is otherwise a soft flag â€” run after each write so the inbox stays bounded with no manual step. `mark_all_read` deliberately clears *every* unread row, not just the loaded page, so the badge it drives reaches zero.

**Schema lives in the model.** The `agent_messages` table is defined by the SQLModel `AgentMessage` in `lib/orm/models/agent_messages.py`: its columns, the `progress` server-default, the mutable read/ack/dismiss timestamps, the `version` supersede counter, and the `MESSAGE_TYPES` severity ladder (`progress Â· note Â· lesson Â· result Â· summary Â· warning Â· blocker`, with `progress` the default) are the authoritative schema for this topic; `severity_rank` on the same module orders that ladder for the push gate. An init/rebuild copy of the table exists in `db/schema.sql` and a migration touching this table must land in both (the schema-drift gotcha), but that DDL is a mirror, not the source of truth here.

**The notification event bus (`lib/agent_messages/events.py`).** Beyond `send_to_user`, a family of *notifiable system events* is declared once in `REGISTRY` and published through a single `emit(kind, â€¦)` that routes to `store.record_message`. Declaring each kind in one table â€” rather than hand-rolling severity, key, deep-link, and guard at each call site â€” makes the full set of things that can reach the inbox **enumerable**: `regin events list` (the CLI in [[regin-cli-architecture]]) and `GET /api/events/kinds` (`web/blueprints/events.py`) both read `events.catalog()`, so "what can reach my inbox" has one source of truth. Each `EventKind` is a frozen dataclass carrying a `kind`, a `severity` (a `MESSAGE_TYPES` value that drives inbox styling and the push gate), a `default_enabled` flag, a human `summary`, and an optional `legacy_flag`. The six declared kinds:

- `permission.pending` (`blocker`, off, legacy `push_permission_events`) â€” the agent is blocked on a permission prompt or `AskUserQuestion`.
- `plan.ready` (`warning`, off, legacy `push_plan_events`) â€” a plan awaits approval on `ExitPlanMode`.
- `topic.suppress` (`warning`, on) â€” a routed topic crossed the fail-rate bar and is proposed for suppression.
- `proposal.ready` (`result`, on) â€” an external agent finished drafting a topic proposal.
- `content.drift` (`warning`, on) â€” a topic's wiki drifted from the code it documents; a refresh proposal is queued.
- `grade.finished` (`note`, off) â€” a session grade landed.

**Enablement precedence** per kind (`is_enabled`): an explicit `settings.agent_messages.events` `{kind: bool}` override wins; else the kind's `legacy_flag` (`permission`/`plan` honor `push_{permission,plan}_events` so existing configs keep working); else the registry `default_enabled`. `emit` returns `None` when the kind is unknown, disabled, or `trace_id` is missing, and is best-effort throughout â€” a notify must never break the producer that emitted it, so `emit`/`resolve` swallow and log every failure. `once=True` skips the write when a live (undismissed) card already exists for `(trace_id, key)`, for producers that must not re-surface a card the user hasn't acted on; `resolve(trace_id, key)` dismisses that card once its condition is handled. Deep-link builders live on the bus â€” `topics_url` (a repo's Topics view), `proposal_url` (`â€¦/topics?tab=proposals&proposal=<id>`, which opens the exact run), and `session_url` (`/trace/sessions/<id>`) â€” their paths mirror `frontend/src/router.js`, so producers don't hard-code UI structure. A card can also be grouped under a *synthetic* trace id that is not a navigable session; those ids are registered in `NON_SESSION_TRACE_IDS` (currently `wiki-debt`) so the inbox card and every push channel omit the dead `/trace/sessions/<id>` link.

**Producers call the bus; each lives in its own subsystem.** Inside this package, `lib/agent_messages/event_notify.py` formats and emits the interaction-required blockers (`permission.pending`, `plan.ready`), de-duping a double-firing prompt (PreToolUse + PermissionRequest) via a DB body-compare (`_already_pushed`) since hooks run as separate processes, and using a stable per-session `msg_key` so the inbox shows one advancing "pending" card rather than a stack. The remaining producers sit in other topics and reach the inbox only through `emit`: `content.drift` from `lib.topics.wiki_debt._notify_drift` (`once=True`, keyed per repo+topic via `content_drift.drift_card_key`, grouped under the synthetic `DRIFT_CARD_TRACE = "wiki-debt"`); `grade.finished` and `topic.suppress` from the graders ([[eval-grading]]); `proposal.ready` from the external-proposal finish path ([[topic-proposal-pipeline]]). One path, no second store.

**Single-emit is a bus invariant** worth calling out because it bites: `emit`'s supersede on `(trace_id, key)` (e.g. `key=proposal-ready:<id>`) dedups the inbox *card*, but `record_message` pushes on *every* write, supersede included â€” so a producer that emits the same event twice re-notifies Feishu/webhook even though only one card renders. Producers that must fire exactly once (notably `proposal.ready`) own that guarantee themselves; the completion-path split that enforces it lives in [[topic-proposal-pipeline]].

**Push fan-out is pluggable.** `lib/agent_messages/push/registry.py` holds the ordered channel list (`_CHANNEL_CLASSES`) and the single `maybe_dispatch` the store calls after every write; `should_dispatch` answers whether any active channel would take a given type. A channel is *active* when configured **and** its `{channel_id}_enabled` switch is on (each defaults on, so the real gate is being configured); a message then fans out to every active channel whose severity gate (`{channel}_min_severity`, default `warning`) it clears. `push/base.py` defines the `PushChannel` base â€” subclasses implement only `is_configured` + `deliver` (which raises on failure rather than returning a status) â€” and `build_push_message`, which projects the stored row once into a provider-neutral `PushMessage`: each app-relative link is resolved to an absolute URL (a bare `/repos/â€¦` isn't clickable in Feishu/Telegram) and a span-anchored `session_url` is built, `None` for a synthetic `NON_SESSION_TRACE_IDS` trace. Every channel shares one `http_post_json` network seam â€” the single point tests monkeypatch. Three channels are registered: `push/webhook.py` (a generic JSON webhook), `push/telegram.py` (a bot chat), and `push/lark.py` (a Lark/Feishu custom-bot incoming webhook, plain-text, HMAC-signed when `lark_secret` is set). `maybe_dispatch` catches each channel's transport error independently so one failing transport never blocks the others, and `_aggregate` collapses the per-channel outcomes into one `sent`/`failed`/`skipped` value for `agent_messages.webhook_status` (or `None` when no channel is active). Registering a new channel is a one-line append to `_CHANNEL_CLASSES`.

**Surface.** The per-session Messages feed and the cross-session inbox routes (`/api/agent-messages/inbox`, `unread-count`, `read`, `read-all`, `<id>/ack|dismiss|pin`) are served by the Flask blueprint `web/blueprints/trace/agent_messages.py` ([[session-trace-design]]); the event catalog reads through `GET /api/events/kinds` in `web/blueprints/events.py`. On the Vue side each row renders in `InboxMessageCard.vue` and the live unread badge is driven by the `useInboxUnread.js` composable; the Inbox page itself is `InboxView.vue` ([[webui-surface]]).
