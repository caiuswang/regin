## Inbox messages & the notification event bus (`agent-messages-inbox`)

Every **inbox message** the human sees flows through one store and, ultimately, one writer. Two kinds of producer feed it: agent-authored `send_to_user` messages, and a declared **notification event bus** for system events the agent didn't type by hand.

**Agent-authored `send_to_user` — the capture path is split on purpose.** The stdio MCP server (`lib/agent_messages/mcp_server.py`) is *session-blind* — it never learns which Claude Code session invoked it — so it does only two things: declare the typed parameter schema (`type`, `title`, `key`, `links`, `supersedes`) and acknowledge the call. The component that *does* know the session is regin's PostToolUse hook: `hook_manager/handlers/post_tool_trace.py` reads the tool input off the landed `mcp__*__send_to_user` span and calls into the store. This keeps the server import-free and instant, and persistence never depends on the server surviving a DB hiccup.

**One writer.** Every write — hook, event bus, web API, or test seed — goes through `lib/agent_messages/store.py::record_message`, so supersede-by-`key`, the 16k body cap (`_BODY_MAX = 16_000`), link normalization, and push dispatch (`push.maybe_dispatch`) all happen in exactly one place. The row model and its `MESSAGE_TYPES` severity ladder (`progress · note · lesson · result · summary · warning · blocker`, with `progress` the default) live in `lib/orm/models/agent_messages.py`; the table plus its `trace_id`/`created_at`/`read_at`/`key` indexes are folded into `db/schema.sql` (mind the schema-drift gotcha — `regin init` builds from `schema.sql`, not Alembic). `store` also owns the keyed-card helpers (`live_keyed_message`, `dismiss_keyed`) that supersedable event cards rely on, plus an opt-in retention sweep (`prune_messages` / `_enforce_retention` past `settings.agent_messages.retention_days`, off by default) that bounds the inbox after each write without a manual step.

**The notification event bus (`lib/agent_messages/events.py`).** Beyond `send_to_user`, a family of *notifiable system events* is declared once in `REGISTRY` and published through a single `emit(kind, …)` that routes to `store.record_message`. Declaring each kind in one table — rather than hand-rolling severity, key, deep-link, and guard at each call site — makes the full set of things that can reach the inbox **enumerable**: `regin events list` (`cli/commands/events.py`) and `GET /api/events/kinds` (`web/blueprints/events.py`) both read `events.catalog()`, so "what can reach my inbox" has one source of truth. Each `EventKind` is a frozen dataclass carrying a `kind`, a `severity` (a `MESSAGE_TYPES` value that drives inbox styling and the push gate), a `default_enabled` flag, a human `summary`, and an optional `legacy_flag`. The six declared kinds:

- `permission.pending` (`blocker`, off, legacy `push_permission_events`) — the agent is blocked on a permission prompt or `AskUserQuestion`.
- `plan.ready` (`warning`, off, legacy `push_plan_events`) — a plan awaits approval on `ExitPlanMode`.
- `topic.suppress` (`warning`, on) — a routed topic crossed the fail-rate bar and is proposed for suppression.
- `proposal.ready` (`result`, on) — an external agent finished drafting a topic proposal.
- `content.drift` (`warning`, on) — a topic's wiki drifted from the code it documents; a refresh proposal is queued.
- `grade.finished` (`note`, off) — a session grade landed.

**Enablement precedence** per kind (`is_enabled`): an explicit `settings.agent_messages.events` `{kind: bool}` override wins; else the kind's `legacy_flag` (`permission`/`plan` honor `push_{permission,plan}_events` so existing configs keep working); else the registry `default_enabled`. `emit` returns `None` when the kind is unknown, disabled, or `trace_id` is missing, and is best-effort throughout — a notify must never break the producer that emitted it, so `emit`/`resolve` swallow and log every failure. `once=True` skips the write when a live (undismissed) card already exists for `(trace_id, key)`, for producers that must not re-surface a card the user hasn't acted on; `resolve(trace_id, key)` dismisses that live card once its condition is handled. Deep-link builders live on the bus — `topics_url` (a repo's Topics view), `proposal_url` (`…/topics?tab=proposals&proposal=<id>`, which opens the exact run), and `session_url` (`/trace/sessions/<id>`) — their route paths mirror `frontend/src/router.js`, so producers don't hard-code UI structure. A card can also be grouped under a *synthetic* trace id that is not a navigable session; those ids are registered in `NON_SESSION_TRACE_IDS` (currently `wiki-debt`) so the inbox card and every push channel omit the dead `/trace/sessions/<id>` link.

**The producers each live in their own subsystem and call the bus.** `lib/agent_messages/event_notify.py` formats and emits the interaction-required blockers (`permission.pending`, `plan.ready`), de-duping a double-firing prompt (PreToolUse + PermissionRequest) via a DB body-compare (`_already_pushed`) since hooks run as separate processes, and using a stable per-session `msg_key` so the inbox shows one advancing "pending" card rather than a stack. `lib/topics/wiki_debt.py::_notify_drift` emits `content.drift` with `once=True`, keyed per repo+topic via `content_drift.drift_card_key`, and `content_drift.resolve_drift_card` clears the card on apply/dismiss. `lib/grader/service.py::_maybe_notify_grade` emits `grade.finished` deep-linked to the trace view (`events.session_url`); `lib/grader/topic_notify.py` emits `topic.suppress` deep-linked to `/memory`. All route through the single `emit`, so each lands as an inbox card *and* fans out through the configured push channels — one path, no second store.

**`proposal.ready` fires from exactly one completion path — on purpose.** `lib/topics/proposal_external.py::notify_proposal_ready` emits `proposal.ready` for a finished external-agent draft. Its action link (`events.proposal_url(repo, proposal_id)`) opens that *specific* proposal run in the Topics review workspace, while its `trace_id` is set to the real drafting-agent Claude Code session — the run's `agent_trace_id`, recovered from the finish command's environment via `lib/session_probe.py::resolve` — so the card's footer "session" link resolves to the actual drafting session rather than the synthetic `topic-proposal-<id>` orchestration wrapper (it falls back to that wrapper id when the drafting session isn't known). Under the notify-on-finish design the authoritative completion is the agent's own `proposal-finish` self-ingest (`lib/topics/proposals/finish.py`), which the server-runner exit may never observe, so *that* path calls `notify_proposal_ready`. The runner exit then **deliberately does not re-notify** on its signalled branch — `_load_signaled_result` returns non-`None`, and `_handle_agent_output` returns the persisted result without calling `_notify_proposal_ready`; only when the agent exits *without* signalling does the runner exit emit the fallback (`_notify_proposal_ready(ctx)` at the end of the non-signalled path). Single-emit is load-bearing: `emit`'s supersede on `(trace_id, key)` (`key=proposal-ready:<id>`) dedups the inbox *card* but not the push fan-out — `record_message` pushes on *every* write, supersede included — so a second emit would re-notify Feishu/webhook even though only one card renders.

**Push fan-out is pluggable.** `lib/agent_messages/push/registry.py` exposes `should_dispatch`/`maybe_dispatch` over a `PushChannel` base (`push/base.py`), which projects the stored row into a provider-neutral `PushMessage` and shares one `http_post_json` network seam. Three concrete channels are registered in `_CHANNEL_CLASSES`: `push/webhook.py` (a generic JSON webhook), `push/telegram.py` (a bot chat), and `push/lark.py` (a Lark/Feishu custom-bot incoming webhook, plain-text, HMAC-signed when `lark_secret` is set). Each has its own severity gate (`{channel}_min_severity`, default `warning`) and is off until configured; a message fans out to *every* configured channel whose gate it clears, and each channel's transport error is caught per channel so one failing transport never blocks the others (`_aggregate` collapses the per-channel outcomes into one `sent`/`failed`/`skipped` status). Registering a new channel is a one-line append to `_CHANNEL_CLASSES`.

**Surface.** The Flask blueprint `web/blueprints/trace/agent_messages.py` serves the per-session feed (`/api/sessions/<id>/agent-messages`) and the cross-session inbox routes (`/api/agent-messages/inbox`, `unread-count`, `read`, `read-all`, `<id>/ack|dismiss|pin`); `mark_all_read` clears every unread row, not just the loaded page. The catalog endpoint `GET /api/events/kinds` lives in `web/blueprints/events.py`. The Vue side is `InboxView.vue`, `InboxMessageCard.vue`, and the `useInboxUnread.js` composable driving the live badge.

*Grounding:* the GitNexus regin index returns no execution-flow processes for the event bus or `notify_proposal_ready` (its graph predates these files), so the event-design (`events.py`, `event_notify.py`, `cli/commands/events.py`, `web/blueprints/events.py`) and producer wiring — including the single-emit `proposal.ready` split across `proposal_external.py` and `proposals/finish.py` — rest on direct source evidence; no call edges were invented.
