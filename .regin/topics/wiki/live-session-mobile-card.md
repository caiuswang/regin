# /live mobile session-tail card

`/live/:id?` (`frontend/src/router.js`, route name `live`) is a mobile-first, single-card view of one session's spans — the phone-sized alternative to the desktop `SessionTraceView`. Design intent lives in `docs/mobile-progress-card-design.md`: a terminal-style recent-spans card with fold-earlier, a live tail, and minimal chrome, explicitly **not** the `send_to_user` message digest that `InboxView` renders. The card also doubles as a remote control: when the session's terminal is bridge-reachable it can steer the running agent and answer its questions (`docs/agent-bridge-design.md`, `docs/bridge-slash-autocomplete-design.md`).

One principle threads the whole view: **state is a server verdict.** The shallow-map summary carries a `phase` rollup and an `agent_phase` map (main + each subagent → its phase); the header and the NOW zone *render* that verdict and never re-derive idleness on the client. See `web/blueprints/trace/sessions.py`'s `_session_phase` / `_main_phase`.

## Shape: one card

`LiveSessionView.vue` renders exactly one `.live-card`:

1. **Header** (`.live-hd`) — a status dot + label driven by the MAIN agent's server phase (`meta.agent_phase.main`) through the `PHASE_STATUS` table: `working` → pulsing `running`, `idle` → steady `idle`, `waiting-permission`/`waiting-input` → amber `waiting`, `inactive-stale` → amber `inactive`, `ended` → grey `✓ finished` (a pre-summary fallback keeps the header honest for the sub-second before the first phase lands). Beside it: an elapsed/last-seen line (`elapsedLabel`) plus optional ended-reason and a `connection lost` hint (two consecutive poll misses). A right cluster of icon buttons — a tasks chip (`done/total` from the server's final task snapshot), an agents button (running-count badge), a raw-terminal peek (only when bridge-reachable), switch-session, and filter/search (a `live-hd-badge` dot when any filter is active). A second header row shows the session goal (tap to expand) and a `repo · model` meta line with a `LiveCtxMeter.vue` live-peak context %. The header gains `live-hd-scrolled` once the tail scrolls past 4px.
2. **Scope bar** (`LiveScopeBar.vue`) — shown only while the tail is scoped to one subagent (see *Per-agent scoping*); the header keeps showing MAIN-session truth.
3. **Fold row** (`.live-fold`) — the first child of the tail, shown only when `hasMoreOlder` and NOT scoped; it displays `earlierCount` not-yet-loaded earlier spans and loads one older turn page on tap, anchoring the scroll position across the prepend (`unfold()` records `scrollHeight`/`scrollTop` before `loadOlder()` and restores the offset after).
4. **Tail** (`.live-tail`, `ref=tailEl`) — the *only* scroll region. Renders a `LiveTailRow.vue` for every row that survives `filterSpans`, plus empty / loading / scope-empty states. A follow-tail rule keeps the view pinned while new spans arrive (`isPinned()` — within 40px of the bottom); when the user has scrolled up, arrivals increment a `↓ N new` chip (`.live-newchip`) that scrolls back down on tap. A blinking caret rides the newest visible row, but only while `mainPhase` is one of `working` / `waiting-permission` / `waiting-input` and never in a subagent scope. A muted `activity unavailable` marker (`degradedRootIds`) renders under any turn whose deep-children fetch was abandoned past the retry cap, so an incomplete tail never reads as a healthy one; it clears itself once a later refresh for that root succeeds.
5. **Queued chips** (`LiveQueuedChips.vue`) — transcript-derived + optimistic just-sent steers (`useQueuedPrompts`), hidden while scoped.
6. **NOW zone** (`LiveNowZone.vue`, `.live-now`) — a sticky footer, at most one state at a time (see below).

Row-detail, full-message, filter, Q&A, agents, tasks, raw-terminal, and session-switcher interactions all happen in **bottom sheets** (`LiveSheet.vue`), positioned absolutely *inside* the card (not a portal) so the tail's scroll survives open/close — `openSheet` saves `tailEl.scrollTop` and `closeSheet` restores it. `LiveSheet` owns the copy-to-clipboard affordance (`live-sheet-copy`) shared by every sheet header. The single `sheet` ref carries a `{ kind, spanId?, agentId? }` where kind is `message` | `detail` | `qa` | `agents` | `agent` | `tasks` | `sessions` | `terminal` | `filter`; the heavier sheet bodies are their own components — `LiveQaSheet.vue` (`qa`), `LiveAgentSheet.vue` (`agents`), `LiveAgentDetail.vue` (`agent`), `LiveTaskSheet.vue` (`tasks`), `LiveTerminalSheet.vue` (`terminal`), and `LiveSessionPicker.vue` (`sessions`).

## Data + poll lifecycle: `useLiveTail`

`frontend/src/composables/useLiveTail.js` is the sole data/lifecycle owner; the view is presentation-only. It rides the same shallow-map pagination the desktop trace view's `useTraceData` uses:

- **Window unit is turn anchors, not spans.** `PAGE_SIZE = 5` pages `/api/sessions/<id>/map?shallow=1&limit=5` by turn-anchor count; activity spans (tool calls, responses, PENDING placeholders) are children of those anchors and stream in separately via `/api/sessions/<id>/spans/<root>/children?deep=1`, fetched with bounded concurrency (`CHILD_CONCURRENCY = 4`). Both endpoints live in `web/blueprints/trace/sessions.py` (`api_session_map`, `api_session_span_children`), shared with the desktop view.
- **Poll cadence**: every `POLL_ACTIVE_MS` (4s) while the session looks active, `POLL_STALE_MS` (15s) otherwise; polling pauses while the tab is hidden and resumes on `visibilitychange`. An `ended` session schedules nothing further — unless a root's children fetch failed during load, in which case polling continues until the retries succeed or hit the give-up cap.
- **`ended` trusts `status` over `ended_at`.** A resumed session (exit tmux → `claude --resume`) keeps a stale `ended_at`; the computed `ended` returns false when `status==='active'` or when `last_seen` runs more than 1s past `ended_at`, so the card doesn't wrongly read `✓ finished` mid-run.
- **`stale` / `active` / `connectionLost`.** `lastSeenAgeMs` is measured server−server (`server_now` vs `last_seen`) so a viewer in another timezone reads the same age; a `status==='active'` session that has ingested nothing for `STALE_FALLBACK_WINDOW_MS` is demoted to `stale` (slows the poll, keeps the NOW zone out of idle). `active` is the shared `isActiveSession` rule minus staleness. Two consecutive mid-session poll failures flip `connectionLost`.
- **Every poll re-fetches the full newest window** (a `limit` cursor, not `after_id`) because `retired_span_ids` — the ids the serve-time merge dropped from the window (a `promptlive-` prompt promoted to its real anchor, a `pending-`/`permreq-` placeholder resolved) — is only computed from that request's raw rows. `pruneStalePending()` is a second safety net: on a poll where every subtree fetch succeeded, any PENDING placeholder absent from that tick's responses is dropped, so a `permreq-`/`pending-` row whose anchor aged out of the 5-anchor window can't pin the NOW zone forever.
- **`refreshSubtrees`** deep-refreshes the trailing `DEEP_REFRESH_TAIL = 3` roots plus any earlier `failedRootIds` (retried, capped at `MAX_ROOT_RETRIES`, so a transient 500 doesn't lose a turn and a persistently-500ing root can't suspend the stale-PENDING sweep forever). A root abandoned past that cap moves into the reactive `degradedRootIds` set — surfaced as the tail's `activity unavailable` marker — rather than being dropped silently as if its turn had no activity.
- **Generation guard (`epoch`)**: `start()` bumps a counter; any async work captured under a stale generation discards its results (`isStale(gen)`), so switching sessions mid-flight can never let a late response from the *old* session merge into the new one.
- **`mergeSpans`** is a keyed, append/update-only merge (pruning is the separate retired-ids path) that prefers non-empty incoming `attributes` over a cached empty shallow placeholder — mirrors `useTraceData.mergeLoadedSpans`. It is exposed so the view can splice lazily-fetched attributes back in (`fetchContent` → `/sessions/<id>/spans/<span>/content`).
- **`appendedSpans`** exposes spans that sort strictly after the previously-last row (a start_time compare, not an id watermark — a rescan-backfilled row can carry a high DB id yet sort mid-list) so the view drives follow-tail / the chip without misreading a filter change as new content.
- **`applySummary`** carries the whole render-truth off the same poll — no side loops: `phase` / `agent_phase` (the state verdict), `bridge_reachable` / `bridge_pane` (the composer gate), `task_list` (final task snapshot), `agent_roster` (whole-session subagent roster, window-independent), `model` / `repo` / `context_pct` (header meta line + ctx meter), `queued_prompts`, and `server_now`. `server_now` (the server's wall-clock at read time) plus a phone-clock stamp (`server_now_at = Date.now()`) let the NOW-zone elapsed anchor to the server clock so a viewer in a different timezone doesn't leak the offset.

## Row semantics: `utils/liveRows.js`

Pure, stateless functions, importing label/color/category primitives from `traceFormatters.js` rather than duplicating them. What's specific to the live card:

- **`rowKind`** splits every span into `msg` (prompt / assistant_response — an eyebrow + up-to-4-line body), `qa` (ask-user-question / permission — its own delicate 2-line mini card), `phase` (a `workflow.phase` marker, rendered as a centered divider band via `phaseBandLabel`, not a tappable row), or `act` (everything else — a faint one-line micro-row whose dot color carries its category).
- **`isQaSpan` / `qaRowModel`** back the `qa` row: `tool.AskUserQuestion`, `permission.request`, `permission.denied`, or any `permreq-` id. A denial arrives as a *separate* `permission.denied` span (the pending `permreq-` placeholder is retired when it lands), so that span is the only surviving record of the outcome — the span NAME is the authoritative denied signal, since the backend never writes decision fields onto `permission.request`. `askRowModel` renders the question glyph, an eyebrow (`Ask user · +N more`), the question, and an outcome (`✓ <chosen>` / `✗ denied` / `✗ answered in chat instead` / `… waiting for your answer`); `permRowModel` renders `⚠ Permission · <tool>`, the `$ <command_preview>`, and granted/denied/waiting.
- **`isSignal`** is the default show/hide tier: an allow-list (`prompt`, `assistant_response`, `subagent.start/stop`, `file.edit`, `plan.edit`, `memory.recall`, `permission.request`, `permission.denied`, `workflow.phase`), plus `tool.*`/`skill.*` prefixes (but never `pre_tool.*`), plus any `rule.check` with findings — errors/denials always show. Anything else (including any *new*, unrecognized span type) is bucketed as system and hidden unless the user flips "Show system spans".
- **`isHotSpan`** keeps a full-saturation dot for edits / failures / denials; every other row renders its category color as muted texture.
- **`humanMain`** renders "what happened", not the raw span type, via a keyed `MAIN_BUILDERS` dispatch + `humanToolMain`: a `tool.Read` on `foo.py` → `Read foo.py`, a `tool.Bash` → a `$ <command_preview>` mono line, a `tool.Agent` / `subagent.stop` → violet agent affordances, `tool.TaskCreate` / `tool.TaskUpdate` → checklist-glyph task rows (`taskEventMain`), and failed/blocked/denied/interrupted tools that SAY so (`toolOutcomeMain` leads with `✗`/`⏹` so a color-only cue never reads as success). Every signal-tier span name must have a phrasing here — raw span-type names must never reach the default view (acceptance 8c). `stripMarkdown` (shared with the NOW zone's clamp) flattens markdown while preserving snake_case identifiers.
- **`filterSpans`** is the single visible-row predicate combining the signal tier, the active category chip, and the search query (same haystack the desktop Terminal searches, via `spanMatchesSearch`). Chip counts are computed over the *same* tier+query-filtered list the rows use so a chip's count always equals its selectable rows.
- **Scoping + summary helpers** also live here: `inScope` / `partitionScope` split the tail by `attributes.agent_id` (main scope = spans without an agent_id plus the three markers; an agent scope = only that agent's spans); `agentStatusLabel` gives one status phrasing for every scoped-agent surface (running · elapsed, finished · duration, `interrupted`, `waiting for input`, `stale · last seen HH:MM`); `taskSummaryOf` rolls the server's final task snapshot into `{total, done, inProgress, open}`.

## NOW zone: `LiveNowZone.vue`

The sticky footer **selects** its state from the server phase and only fills **content** from the already-loaded tail — it never re-derives idleness. `PHASE_STATE` maps each phase: `ended` → finished, `inactive-stale` → inactive, `waiting-permission` → permission (else response), `waiting-input` → question (else response), `idle` → idle, `working` → the newest pending tool/prompt content else response. A pre-summary fallback uses legacy pending-priority but never invents idle or a stale finished. Permission and question get the amber `live-now-attention` treatment; question surfaces the question text + option count with an `options ▾` that opens the Q&A sheet.

- **Content by priority.** `findLastSpan` reads the NEWEST pending placeholder per kind — permission (`permreq-*`) › question (`pending-*` AskUserQuestion) › tool (other `pending-*`) › live prompt (`promptlive-*`) — so a second question cleanly replaces the first once the poll retires the earlier placeholder.
- **`showWorking`** gates the spinner / "working…" copy on the *client* `active` gate too, so an `inactive` footer never contradicts an `inactive` header.
- **Timezone-safe elapsed.** The elapsed for a PENDING permission/question/tool comes from the one shared formula, `agentElapsedSeconds` in `composables/useAgentElapsed.js`: `(server_now − span.start_time) + (phone_now − server_now_at)` — both leading terms server-local, so a phone in another timezone doesn't leak the offset. The 1s tick runs only while a tool/permission/question state shows it. `STALE_PENDING_MS = 45000` suppresses the clock when fresher activity exists beyond the pending span (timing a subagent parent / orphaned placeholder, not the current step). `fmtElapsedSeconds` rolls seconds over to `8m09s` / `1h05m`.
- **Idle sub-note.** An idle MAIN agent can still have background subagents working or blocked — invisible to the main projection — so the idle state surfaces an `◈ N agents running · M waiting` link (roster counts) that opens the agents sheet.
- **Scoped variant.** When `scopeAgent` is set the zone shows that agent's `agentStatusLabel` + a `back to main ↩` action, and the composer is hidden (the bridge reaches only the main agent).

## Bridge composer + answerable Q&A

When `bridge_reachable` is true the NOW zone mounts `LiveComposer.vue` (a child, so all async send state stays out of the projection): a full-width composer in idle/inactive (starts the next turn) and a compact steer variant in the working `response`/`tool`/`prompt` states (queues into the running turn); absent in question/permission/finished, in a subagent scope, or when unreachable. It POSTs the typed text to the web-JWT proxy `/api/sessions/<id>/bridge-send` (`web/blueprints/bridge.py`), which is JWT + `require_editor`-gated (viewers get 403) and delivers in-process into the session's registered tmux pane; a raw control key (Escape-to-interrupt) goes through the sibling `/bridge-key`. Delivery is confirmed only when the poll returns the real `promptlive-`/`prompt` span — no client-stamped row is ever appended (the just-sent steer rides `useQueuedPrompts` as an optimistic chip until then). The draft is a `v-model:draft` owned by `LiveNowZone` (which never unmounts), so text typed mid-draft survives a one-poll reachability blip that unmounts the composer, and resets only on a session switch. `LiveCommandMenu.vue` + `useSlashCommands.js` provide `/command`/skill autocomplete over `/api/sessions/<id>/bridge-commands` (design in `docs/bridge-slash-autocomplete-design.md`).

`LiveQaSheet.vue` is the full Q&A detail behind a qa row and the NOW zone's `options ▾`. For a PENDING ask over a reachable bridge whose questions are **all single-select** (`singleSelectOnly`, `canAnswer`) it delegates to an interactive operator surface, routed by question count (`isMulti`): `LiveQaAnswer.vue` for a single question (pick an option with an optional note, type your own via the TUI's "Type something." entry, or "Chat about this" — each behind a select→confirm gate so a mis-tap can't answer the live agent irreversibly, its three verbs mapped in `buildPayload`), or `LiveQaAnswerMulti.vue` for several questions (a Prev/Next stepper that collects one single-select answer per question via `payloadFor`, revisable in-card, and delivers nothing until "Send all"/`sendAll`). Both POST `/bridge-answer`: the single answerer sends `{option_index, text?, label?, chat?}`; the multi answerer sends an ordered `answers` array that the backend walks tab-by-tab, each entry carrying a `confirm_text` (its question) so the backend refuses if a failed advance left an earlier question focused — collect-then-send is what makes "go back and change an answer" safe. Delivery is confirmed by the proxy's `delivered` flag before the sheet closes. Both answerers share `LiveQaGrowInput.vue` for their auto-growing text fields. An ask containing a **multi-select** question (`hasMultiSelect`) stays read-only with an "answer it in the agent terminal" note — a multi-select TUI needs per-option toggles the bridge can't drive blindly — as does any unreachable or already-completed span. A permission span renders its requested command + granted/denied/waiting outcome (same `permission.denied`-is-authoritative rule as `permRowModel`). `LiveTerminalSheet.vue` is the read-only raw-terminal peek: a one-shot `GET /bridge-screen` (a ~200-line scrollback page, refreshable, never a live stream) whose already-`ansi_html.convert()`-ed markup opens scrolled to its own bottom so the live status line shows first — the one surface for harness-only interactions (slash menus, the box-drawn status line) that never produce a span.

## Per-agent scoping + roster

`useLiveAgents` builds the whole-session agent roster off the server's `agent_roster` (window-independent — loaded-tail markers alone would drop agents that aged out); the server owns classification (running / finished / waiting / interrupted / stale). The agents button opens `LiveAgentSheet.vue`; tapping an agent opens `LiveAgentDetail.vue` (re-resolved live off the roster) or **scopes** the tail via `useLiveScope`, which re-partitions the tail to that subagent's spans (auto-paging older windows into view on first entry), shows `LiveScopeBar.vue` + a scoped NOW zone, and restores the exact main-tail scroll on exit. Every scoped surface shares the `agentElapsedSeconds` clock from `useAgentElapsed.js`.

## Session switcher: `LiveSessionPicker.vue`

Opened from the header's list icon into a `sessions` bottom sheet (the sheet body is `v-if`, so the picker remounts fresh on every open and fetches once per mount). It fetches `GET /sessions?kind=real&size=20`, sorts active-first then newest-`last_seen`-first, and emits `select`; `onPickSession` then `router.push('/live/' + trace_id)`. `kind=real` excludes Playwright/test fixture sessions from the list, but a direct `/live/<test-session-id>` deep link still resolves because `useLiveTail.resolveSessionRow` looks up `kind=all`. Each row's active dot and the active-first sort run through `utils/sessionActivity.js`'s `isActiveWithClock(row, clock)`, which ages `last_seen` server−server against a `{ local: server_now, utc: server_now_utc, atMs }` clock envelope built from that list response — so a viewer in another timezone reads the same activity as the host. `isActiveWithClock` composes the shared `isActiveSession` rule (also used by the desktop `SessionsView`) with `serverAgeMs`; `parseLocalIso` in the same file parses the backend's naive local-time ISO strings that native `Date()` would mis-read as UTC.

## Testing

`frontend/tests/live-card.spec.js` is organized by acceptance criterion and slice: the 375px invariant, default signal filter, fold-row states, poll lifecycle, filter sheet, message-first hierarchy, bottom sheets, incremental tail arrival, the NOW-zone state priority + elapsed rollover, the idle state + bridge composer / send lifecycle, the terminal peek sheet, answering a pending single-question ask from the QA sheet, answering a pending multi-question ask (the Prev/Next stepper collecting every answer + "Send all" delivering in order, plus the multi-select-question read-only fallback), the header tasks chip + tasks sheet, the header agents button + agents sheet, differentiated tail rows, the ctx meter, per-agent span scoping + auto-page-on-entry, and agent lifecycle (interrupted / stale / resumed). `frontend/tests/live-card-phase-real.spec.js` drives the phase-driven header/NOW-zone state against real server-phase summaries; `frontend/tests/context-live-peak.spec.js` covers the `LiveCtxMeter` live-peak context %; `frontend/tests/trace-live-prompt-handoff.spec.js` covers the optimistic-steer → real-`prompt`-span handoff; `frontend/tests/workflow-live-phases.spec.js` covers the `workflow.phase` divider bands. Bridge scenarios patch the map response in-flight and stub the `/bridge-send`/`/bridge-answer` proxies, since the test server runs with the bridge disabled. `frontend/tests/live-picker.spec.js` covers the session switcher (including the no-cross-session-leak of an optimistic steer); `frontend/tests/responsive.spec.js` covers the broader mobile/responsive layout pass.

## Relationship to the desktop trace view

The live card is a sibling read path to `SessionTraceView`'s **conversation** mode, not a replacement — see [[session-trace-design]] and [[trace-merge-reconcile]] for the shared shallow-map/serve-time-merge model both views build on, and [[webui-surface]] for where `/live` sits in the SPA route/nav map.
