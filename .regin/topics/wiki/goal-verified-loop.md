## Loop-engineering: goal preflight → verified build → feedback (`goal-verified-loop`)

The regin-side halves of the `/goal-verified` workflow. The idea: pin a *falsifiable* bar **before** the agent builds, so verification checks against something concrete instead of the agent grading its own homework.

**Preflight (front half, `lib/goal_preflight.py`).** A deterministic, offline kernel — no embeddings, no LLM. `build_roadmap(goal)` returns a `Roadmap` whose core is the universal hard-gate floor `BASE_GATES`: the existing test suite stays green, and an independent fresh-context reviewer checked the diff (`/code-review high`). That floor mirrors what the loop always requires, so the roadmap never invents a standard. Per-area machine gates (pytest / radon / vite / playwright), convention skills, and reference components are sourced *outside* this kernel — the file-keyed convention table in `CLAUDE.local.md` (read by area before editing) and the real target files (or, in the tree-nav arm, the topic leaf's source refs). The one irreducibly generative step — turning the gates into concrete, falsifiable acceptance items at 0/1/N — belongs to the consuming skill; `render_markdown` leaves the Acceptance checklist as an explicit prompt. An opt-in lessons leg (`build_roadmap(with_lessons=True)`, `--with-lessons`) runs `recall_lessons` in FTS mode (lexical BM25 on the goal text, pre-code) and returns a short menu of `{id, title, snippet}`; it is **off by default at the CLI** (demoted 2026-06 to ~22% engagement) with structure-first `regin memory recall-for-task` preferred, retained for A/B. `record_offered(--session-id …)` logs the offered ids as an injection event *without* bumping `recall_count`, so the engagement denominator is automatic even when `goal feedback` is skipped; `reinforce=False` on recall is load-bearing for the same reason.

**Feedback (back half, `lib/goal_feedback.py`).** After a verified run, `record_outcome` writes back to the shared `lib.memory` store, reusing `remember`/`reinforce` only (no new table): (1) **engagement** — a lesson that made it into the approved acceptance checklist (`--included`) is reinforced; offered-but-unused ids are left untouched so reflect's `_decay_chronically_ignored` handles decay. Human inclusion is a higher-precision 'did this help' signal than the trace-referent heuristic in `lib.memory.feedback`. (2) **new lessons** — each FAILED acceptance item (`--fail`, phrased as a transferable rule by the skill) becomes a `lesson` memory tagged with the area plus `FAIL_TAG = goal-verified-fail`. (3) **topic filing** — `--topic <node-id>` files each new failure-lesson under an authoritative topic node the agent resolved from its tree walk (`source = goal-feedback`); resolution is exact-id-or-slashed-leaf only, with **no fuzzy keyword fallback** (a wrong authoritative link is worse than none), and a `none`/`-` sentinel files the lesson unbound on purpose.

The build + verify halves themselves live in `.claude/skills/goal-verified/SKILL.md` and its tree-nav arm `.claude/skills/goal-verified-treenav/SKILL.md` (cited as docs).
